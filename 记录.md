现在 paper 验收这一块基本收口了，剩下的是生产化能力，不是刚才那批测试。

  按优先级看，主要还有这些：

  1. 更广 broker reconciliation / production audit
      - 更长窗口、更多资产、更多 broker 的 open orders / executions / cash / position reconciliation 长跑。
      - 验证订单恢复、告警、reconciliation 审计在真实 broker 场景下稳定。
      - 这是最该先开的 plan。

  2. IBKR 完整合约/多资产 reconciliation
      - 现在 IBKR paper Gateway ReadOnly / AutoRun / 3 轮 Soak 已过。
      - 但还不是完整合约、多资产、长时间窗口的 reconciliation 覆盖。
      - 还缺 liquidation / open-interest / contract metadata 等生产字段接入。

  3. 非 IBKR broker reported cash/account snapshot
      - 当前 IBKR paper snapshot/reconciliation 边界已接。
      - 还缺其它 broker 的 reported cash/account snapshot 调度和联调。

  4. Reference-data 生产运维
      - reference-data 限流退避。
      - 陈旧数据告警。
      - ingestion SLA。
      - open interest / liquidation buffer 等交易所参考数据采集。

  5. 生产级 RBAC / config governance
      - 真实身份认证。
      - RBAC。
      - 多环境权限矩阵。
      - 多人审批队列。
      - 生产变更报表。

  6. 托管式外部生产日志采集 agent
      - 现在已有 HTTP NDJSON shipper、system_logs、retention、查询/export。
      - 但还不是完整托管式生产采集 agent。

  7. 大规模历史数据湖 schema
      - tick、order book、funding rate、open interest、fundamentals、corporate actions 的分区 schema。
      - ingestion SLA 和陈旧数据告警。
      - 这个优先级可以放后，等 paper/live 闭环更稳。

  我建议下一步先做 production reconciliation / contract metadata hardening plan。这条最靠近交易正确性，能把“paper 通过”推进到“生产前可信”。
