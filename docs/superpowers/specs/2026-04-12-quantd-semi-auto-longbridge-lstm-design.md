# Quantd 半自动实盘系统设计（Longbridge + LSTM）

日期：2026-04-12  
状态：Draft  
范围：`quantd` 运行时收敛、`lstm-service` 跑通、美股单账户半自动交易、一组标的批量调度、Longbridge 实盘执行

---

## 1. 目标与边界

本阶段目标不是继续扩展一个通用量化平台，而是交付一套可控的半自动实盘系统：

- 单账户
- 先美股，后续再扩港股
- 单一 LSTM 打分策略
- 一组标的批量运行
- 定时批处理，不做高频盘口交易
- 通过 Longbridge 真实下单
- 人工盯盘，系统支持快速暂停、降级和审计

本阶段明确采用：

- 方案 1：运行时收敛成单体实盘机
- 方案 3：训练/研究与交易执行边界分离

因此首期系统定义为：

- `quantd`：唯一运行时交易核心
- `lstm-service`：独立模型服务，负责训练和推理

---

## 2. 当前项目现状判断

当前仓库同时存在两种尺度不同的实现：

1. 一条较清晰的 `quantd` 主链路  
   `ingest -> strategy -> risk -> exec -> db -> api`

2. 一批范围明显更大的平台化模块  
   包括 `strategy`、`risk`、`exec`、`marketdata`、`infra` 中大量面向通用平台的扩展能力

结合近期提交记录与根文档，当前真正贴近业务目标的是：

- `crates/quantd`
- `crates/pipeline`
- `crates/api`
- `crates/db`
- `crates/longbridge_adapters`

当前真正的主要问题不是“缺一个功能”，而是：

- 运行时交易系统和研究平台目标混在一起
- 主链路仍停留在单标的单次 tick 思维
- 风控还停留在 MVP 级别
- Longbridge 已接通方向，但账户一致性、对账、停机和幂等控制不足
- `lstm-service` 还没跑通，不能被当成稳定依赖

---

## 3. 核心设计原则

### 3.1 运行时最小闭环优先

首期只为以下闭环负责：

`稳定行情 -> 稳定模型分数 -> 稳定组合决策 -> 严格风控 -> 稳定下单 -> 可追溯`

### 3.2 训练与执行解耦

- Python `lstm-service` 负责训练与推理
- Rust `quantd` 只消费 `/predict`
- 首期不依赖在线训练、在线回测、复杂模型管理

### 3.3 批量组合决策优先于单标的信号

当前 `pipeline` 偏向“单个标的产生一个信号就下单”。  
首期目标是一组标的的 ML 排名策略，因此必须引入批量评分和组合决策层。

### 3.4 先做硬风控，再做聪明风控

首期风控只做可解释、可拒绝、可审计的硬规则，不引入复杂动态调节逻辑。

### 3.5 无法确认状态时宁可不交易

以下情况默认不下单：

- 模型服务不可用
- 数据不足
- 券商状态异常
- 对账不一致
- 配置不完整

---

## 4. 首期系统架构

### 4.1 运行时组件

`quantd` 内部运行时只保留以下组件：

1. `Scheduler`
2. `Batch Ingest`
3. `Batch Score`
4. `Portfolio Decision`
5. `Risk Gate`
6. `Execution`
7. `Reconcile`
8. `Ops Control`

### 4.2 数据流

```
Scheduler
  -> Batch Ingest
  -> Batch Score
  -> Portfolio Decision
  -> Risk Gate
  -> Order Plan
  -> Execution
  -> Reconcile
  -> Audit / Ops View
```

### 4.3 组件职责

#### Scheduler

- 固定周期触发一轮任务
- 推荐分钟级，不做 tick 级高频
- 同时保证上一轮未完成时不会重入

#### Batch Ingest

- 对标的池逐个拉取最新 K 线
- 统一写入 `bars`
- 行情写入失败要逐标的记录，不可整轮静默忽略

#### Batch Score

- 读取每个标的最近 `lookback` 根 K 线
- 调用 `lstm-service /predict`
- 返回 `score`、`confidence`、`side`
- 单标的失败不拖垮整轮

#### Portfolio Decision

- 汇总整组标的评分
- 进行排名、过滤、预算分配
- 生成理论目标持仓
- 将目标持仓映射为本轮 `OrderPlan`

#### Risk Gate

- 校验本轮调仓计划是否合法
- 拒绝不满足限制的委托
- 生成拒绝原因并持久化

#### Execution

- 通过 Longbridge 发起真实委托
- 首期仅支持最简单基础单型
- 每次提交必须具备幂等与审计信息

#### Reconcile

- 周期性同步券商现金、持仓、未完成订单
- 本地账本与券商状态不一致时进入降级模式

#### Ops Control

- 查看当前模式
- 暂停/恢复自动下单
- 开启只观察模式
- 紧急停机
- 标的池 allowlist 管理

## 5. LSTM 服务设计

### 5.1 角色定位

`lstm-service` 是首期交付的一部分，但其目标不是成为完整研究平台。  
首期只要求它成为一个稳定可调用的模型服务。

### 5.2 首期必须跑通的能力

- 本地可启动
- 依赖安装清晰
- `GET /health` 可用
- `POST /predict` 可用
- 至少支持一个美股样本标的
- 能加载已有模型或完成最小训练
- 推理输入输出格式稳定
- 对超时、模型缺失、数据不足给出明确错误

### 5.3 首期不强求的能力

- 完整在线训练平台
- 多模型治理
- 多 provider
- 在线回测接口
- 完整特征诊断平台

### 5.4 与 quantd 的契约

`quantd` 首期只依赖：

- `GET /health`
- `POST /predict`

推荐的 `POST /predict` 结构：

```json
{
  "symbol": "AAPL.US",
  "model_type": "alstm",
  "bars": [
    {
      "ts_ms": 1700000000000,
      "open": 180.0,
      "high": 182.0,
      "low": 179.0,
      "close": 181.0,
      "volume": 50000000
    }
  ]
}
```

响应：

```json
{
  "score": 0.73,
  "side": "buy",
  "confidence": 0.81
}
```

## 6. quantd 运行时收敛设计

### 6.1 保留并强化的模块

- `crates/quantd`
- `crates/pipeline`
- `crates/api`
- `crates/db`
- `crates/longbridge_adapters`

### 6.2 降级为后续方向的模块

以下能力不作为首期运行时建设重点：

- `risk` 作为独立复杂子系统
- `infra` 的微服务化和大型运维体系
- `marketdata` 的重型数据平台能力
- `strategy` 中复杂优化、回测、热重载继续扩展
- 高级订单类型与复杂执行算法

### 6.3 关键新增运行对象

为支持“一组标的 + 单一 ML 策略”，建议引入以下运行对象：

- `UniverseRunner`
- `ScoredSymbol`
- `PortfolioTarget`
- `OrderPlan`
- `ReconcileSnapshot`

推荐职责：

- `UniverseRunner`：驱动单轮全标的调度
- `ScoredSymbol`：单标的评分结果与状态
- `PortfolioTarget`：理论目标持仓
- `OrderPlan`：实际下单计划
- `ReconcileSnapshot`：券商与本地状态快照

### 6.4 对当前 pipeline 的调整方向

当前 `run_one_tick_for_venue` 可以保留为底层单标的能力，但不应直接作为一组标的半自动交易的顶层运行模型。  
顶层运行模型应改为：

`run_one_cycle_for_universe`

其流程为：

1. 对标的池完成 ingest
2. 批量调用模型服务评分
3. 统一做排名和预算分配
4. 生成组合级调仓计划
5. 再逐单进入执行与审计

## 7. 风控设计

### 7.1 首期必须落地的硬风控

- 单票最大资金占比
- 最大同时持仓数
- 每轮最大新增仓位
- 每日最大成交额
- 每日最大亏损阈值
- 同标的最小重复下单间隔
- 模型最低置信度或分数阈值
- 数据完整性检查
- 券商账户状态检查
- 全局 `panic_stop`

### 7.2 风控原则

- 风控未通过必须拒绝交易
- 每次拒绝必须写入明确 reason
- 风控拒绝不应被自动重试绕过
- 默认拒绝优先于默认放行

### 7.3 不纳入首期的复杂风控

- 动态 VaR
- 复杂相关性约束
- 蒙特卡洛压力测试在线执行
- 高级组合优化器

这些能力可以保留在研究侧或后续阶段，但不阻塞首期半自动运行。

## 8. Longbridge 实盘执行设计

### 8.1 首期执行方式

- 单账户
- 美股
- 最基础单型
- 不做 TWAP / VWAP / Iceberg
- 所有订单必须可追溯

### 8.2 必须补足的实盘能力

- 启动时同步券商现金、持仓、未完成订单
- 本地下单状态与券商状态周期性对账
- 下单幂等
- 委托失败不做扫单式自动重试
- 手工暂停后不继续自动下单
- 服务重启后不因历史信号重复委托

### 8.3 降级机制

出现以下情况时切换为 `observe_only` 或 `degraded`：

- 对账不一致
- 券商连接异常
- 账户状态异常
- 关键配置缺失

---

## 9. 运维控制面设计

半自动运行必须先具备控制面，不依赖人工改数据库或重启进程处理所有情况。

首期至少提供以下控制能力：

- 查看当前运行模式：`enabled` / `observe_only` / `paper_only` / `degraded`
- 查看标的池
- 查看当前策略配置
- 查看最近一轮评分结果
- 查看最近一轮订单计划
- 查看最近风控拒绝
- 暂停/恢复自动下单
- 切换只平不开
- 紧急停机

---

## 10. 审计与可观测性

首期必须能回答“为什么这笔单会出去”。

因此至少保留以下审计事件：

- 每轮调度开始/结束
- 每个标的的评分结果
- 每轮组合决策摘要
- 每个风控拒绝原因
- 每笔下单请求与券商响应
- 每次对账结果
- 每次模式切换与人工操作

日志要求遵循现有结构化 tracing 规范。

---

## 11. 失败处理与降级策略

### 11.1 模型服务失败

- `lstm-service` 不可用：整轮不下单，写审计，系统保持存活
- 单标的数据不足：跳过该标的，不影响其它标的
- 模型缺失：该标的不交易

### 11.2 券商执行失败

- 下单失败：记录失败并停止自动重试扫单
- 券商返回异常：保留原始错误并进入审计

### 11.3 状态不一致

- 对账不一致：进入 `observe_only`
- 配置错误：启动失败，不允许带病运行

## 12. 分阶段实施计划

### 阶段 0：冻结范围

首期只做：

- 单账户
- 美股
- 单一 LSTM 策略
- 一组标的
- 半自动运行
- Longbridge 实盘执行

同时暂停继续扩张以下方向：

- 大型 `infra`
- 独立复杂 `risk`
- 重型 `marketdata`
- 多市场、多账户、多策略并发
- 高级订单类型

### 阶段 1：先跑通 `lstm-service`

这是当前第一优先级。

验收标准：

- 服务本地可启动
- `GET /health` 正常
- `POST /predict` 正常
- 至少支持一个美股样本标的
- Rust 可稳定联调

### 阶段 2：建立批量标的调度层

将顶层运行从“单标的一次 tick”升级为“整组标的一轮调度”：

- 批量 ingest
- 批量 score
- 排名筛选
- 预算分配
- 生成组合级订单计划

### 阶段 3：补齐实盘硬风控

落地单票、总仓、日内额度、冷却时间、账户状态、全局停机等硬规则。

### 阶段 4：补齐 Longbridge 实盘链路

补全：

- 启动同步
- 周期对账
- 幂等
- 停机控制
- 重启一致性

### 阶段 5：控制面与审计

提供运行模式管理、评分结果查看、订单计划查看、风控拒绝查看、暂停与紧急停机。

### 阶段 6：渐进式上线

建议顺序：

1. `observe_only`
2. `paper`
3. `tiny live`
4. `limited live`
5. 再考虑港股扩展

## 13. 风险与回滚思路

### 13.1 主要风险

- `lstm-service` 依赖过重、训练链路难以稳定
- 运行时仍延续单标的思维，导致批量组合交易失控
- 实盘状态与本地状态漂移
- 缺少控制面导致异常时只能人工修改库或重启

### 13.2 回滚策略

- 模型服务异常时，切回 `observe_only`
- 实盘链路异常时，切回 `paper_only`
- 对账异常时，停止自动下单
- 新增批量调度层若不稳定，可临时回退到只评分不下单

---

## 14. 结论

当前项目最大的问题不是技术实现能力不足，而是规划尺度和首期目标不匹配。  
如果继续按“大而全平台”推进，很可能得到一个模块很多、文档很多、crate 很多，但不敢挂半自动实盘的系统。

正确方向是：

- 运行时收敛成单体实盘机
- 训练与研究从运行时剥离
- 先把 `lstm-service` 跑通
- 把顶层模型从“单标的单次下单”升级为“组合级批量调度”
- 在自动下单前补齐硬风控、对账、控制面和审计

只有这样，系统才会从“能演示”进入“能半自动实盘运行”。
