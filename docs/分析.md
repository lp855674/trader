   ## 3. 核心架构原则实现度

   ### 3.1 架构原则遵循情况
   ✅ 策略不直接下单（通过Algorithm Framework）
   ✅ 多运行模式共用核心逻辑
   ✅ 事件驱动（有event_store表）
   ⚠️  市场规则插件化（有market_rules crate，但配置表缺失）
   ✅ OMS是订单核心
   ⚠️  存储分层（SQLite就绪，Parquet未实现）

   ### 3.2 关键设计缺失
   ❌ Parquet历史数据（data/parquet/目录不存在）
   ❌ 复杂市场规则配置（fee_rules等表缺失）
   ❌ 订单事件溯源（order_events表缺失）
   ❌ 风控事件记录（risk_events表缺失）
   ❌ 数字货币合约支持（crypto_positions表缺失）

   ## 4. 实际可行性评估

   ### 4.1 MVP可行性：✅ 可行
   **当前实现足以支持：**
   - 股票回测（A股、港股、美股）
   - 简单的数字货币现货交易
   - 基本的Portfolio管理
   - 简单的风控（在代码层面）
   - 模拟交易（IBKR、Binance）

   **简化设计的合理性：**
   1. **用event_store替代多个事件表**
      - 优点：灵活、易扩展、支持事件溯源
      - 缺点：查询效率可能较低

   2. **市场规则在代码中硬编码**
      - 优点：开发速度快
      - 缺点：规则变更需要重新编译

   3. **没有Parquet支持**
      - 影响：无法高效处理海量历史数据
      - 缓解：SQLite足够回测用（小规模）

   ### 4.2 缺失功能的影响度

   #### 高影响（影响核心功能）
   ❌ **order_events** - 影响订单调试和故障排查
   ❌ **crypto_positions** - 无法正确处理合约持仓（杠杆、资金费率）
   ❌ **funding_rates** - 永续合约PnL计算不准确

   #### 中影响（影响完整性）
   ⚠️  **risk_events** - 风控决策无审计记录
   ⚠️  **market_calendars/trading_sessions** - 交易时间校验依赖硬编码
   ⚠️  **fee_rules** - 手续费只能用固定比例

   #### 低影响（可以延后）
   △ **insights/portfolio_targets** - 主要用于分析，不影响交易
   △ **configs** - 配置管理可以用文件系统
   △ **Parquet** - 小规模回测不需要

   ## 5. 为什么8张表就能工作？

   ### 5.1 设计简化的巧妙之处

   1. **event_store替代多个事件表**
      ```
      order_events + risk_events + system_logs → event_store
      ```
      所有事件以JSON存储，通过category字段区分

   2. **市场规则内置在market_rules crate**
      ```
      fee_rules + lot_size_rules + price_limit_rules → 代码实现
      ```
      规则逻辑在Rust代码中，不需要数据库表

   3. **positions表统一处理**
      ```
      positions + crypto_positions → positions
      ```
      通过asset_class字段区分类型

   4. **配置使用文件系统**
      ```
      configs表 → configs/目录 + TOML文件
      ```
      20多个config文件证明这个方案可行

   5. **历史数据暂时不需要**
      ```
      Parquet历史行情 → 测试时使用fake data
      ```
      回测使用代码生成的测试数据

   ### 5.2 实际数据验证
   ```
   trader.sqlite: 1.7MB
   - 1个strategy_run
   - 1个order
   - 1个fill
   - 2个positions
   ```
   说明系统确实在运行，只是规模很小

   ## 6. 存在的风险

   ### 6.1 技术债务
   1. **可扩展性受限**
      - 市场规则硬编码，新市场需要改代码
      - 没有历史数据支持，无法做大规模回测

   2. **调试困难**
      - 缺少order_events，订单问题排查困难
      - 缺少risk_events，风控决策不可追溯

   3. **数字货币合约支持不完整**
      - 缺少crypto_positions，无法正确处理杠杆
      - 缺少funding_rates，PnL计算有误差

   ### 6.2 扩展性问题
   当需要支持以下场景时会遇到瓶颈：
   - 大规模历史数据回测（需要Parquet）
   - 复杂的市场规则配置（需要规则表）
   - 多账户、多策略并行（需要更好的审计）
   - 数字货币合约交易（需要crypto相关表）

   ## 7. 结论

   **缺点：**
   ❌ 数据库表简化过度（24 → 8）
   ❌ 数字货币合约支持不完整
   ❌ 缺少历史数据存储（Parquet）
   ❌ 订单事件溯源缺失
