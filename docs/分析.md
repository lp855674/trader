本文核对范围：

- 设计文档：`tech.md`、`docs/tech/storage.md`、`docs/tech/api.md`、`docs/tech/data.md`、`docs/tech/feature_store.md`、`docs/roadmap.md`

## 结论

当前实现已经从 8 表 MVP schema 前进到覆盖 24 张目标 SQLite 表、带审计投影、市场规则参考表、合约持仓/funding storage boundary 及 API/CLI 只读查询面、paper 合约 runtime accounting、Binance position reconciliation 测试、IBKR broker account/position snapshot adapter、Binance/Yahoo reference-data ingestion、市场元数据/公司行动只读查询面、paper 现金/持仓快照、live 启动 baseline 现金快照、可配置 fake broker cash/position snapshot 调度、IBKR paper Gateway 由 live API 配置注入的 snapshot/reconciliation 边界、broker open orders/executions 启动恢复同步、恢复成功的 `broker.order.recovered` 审计投影、恢复失败的 run failed 边界、未知远端 open order 默认阻断启动并标记 run failed、显式 `[live.startup_recovery] unmatched_open_orders = "warn_only"` 兼容策略、paper 示例配置显式写出默认 `fail` 策略、startup recovery 默认 fail / warn_only API smoke 覆盖、cash/position reconciliation drift 风险事件投影、run-scoped snapshot/reconciliation API/CLI 查询、cross-run / run-scoped reconciliation drift audit API/CLI 查询、reconciliation alert summary API/CLI 查询、live reconciliation gate 的 config/CLI/script/runbook 和 fresh multi-broker read-only gate evidence、API 和 CLI 运行配置快照、config version/state/diff/rollback lifecycle API/CLI、production config 独立审批发布约束、config release/run-version binding/audit 查询面、runtime/system log 过滤、retention purge / startup cleanup / server background cleanup 查询面、drop-metric ops readback、带 retry/backoff 和 HMAC 签名的 HTTP NDJSON collector shipper 和 `[logging]` 配置默认值，以及 `scripts/ops-smoke.ps1` 覆盖的 live run + config governance 运维端到端冒烟链路的本地 MVP / paper 验证系统。IBKR paper production reconciliation 已有 6 轮 read-only、3 轮 paper order-submit、30 轮 extended read-only 验收证据，总计 39 次 audits 且 drift/stale counters 全为 0；Binance paper run/soak 也已通过，并且 fresh read-only IBKR/Binance evidence 已通过 reconciliation gate。它仍不是完整生产级多市场交易系统，因为真实资金 live-account、更多 broker、多资产 filled-order reconciliation、生产级 RBAC/多环境权限矩阵、托管式外部生产日志采集 agent、reference-data 生产限流/陈旧告警和多人审批队列尚未全部接线。

## Production Reconciliation / Contract Metadata

Paper-account production reconciliation is accepted for the current IBKR gate. The hardening plan is `docs/superpowers/plans/2026-07-07-production-reconciliation-contract-metadata-hardening.md`; the accepted evidence is summarized in `docs/production-reconciliation-acceptance-summary.md`.

Remaining limits after this paper-account gate:
- IBKR contract metadata coverage depends on fields returned by Gateway for each security type.
- Liquidation price and open interest are partial until each broker/reference-data source reports them.
- Non-IBKR broker account snapshot scheduling remains a follow-up once the shared broker boundary is stable.

最初“8 张 SQLite 表能工作”的判断对早期 MVP 是成立的；当前修复后，系统已新增结构化审计查询投影、可版本化市场规则表、合约持仓/funding storage boundary、paper runtime 写入路径、CLI/API 只读查询面，以及 reference-data storage boundary 的只读查询面。这个取舍对本地回测、Replay、simulated paper、Binance Spot Testnet paper、IBKR paper 验证仍然合理；对生产级合约交易、完整市场元数据更新、机构级审批和变更审计仍然不足。

旧分析中的核心误判：

- Parquet 不是未实现；当前已有 bars Parquet 读写和 feature Parquet 读写。
- 测试数量和失败率不应写死；旧报告中的固定测试统计不可信。
- `positions + crypto_positions 通过 asset_class 统一` 不准确；当前 `positions` 表没有 `asset_class` 字段。
- `订单事件溯源缺失` 说法过重；当前有通用 `event_store`，也有 `order_events` 等结构化查询投影。

### Parquet 当前实现

Parquet 已经实现，不应再写成缺失项：

- `crates/data/src/parquet.rs` 提供 bars Parquet 读写。
- `crates/feature_store/src/feature_store.rs` 提供 `ParquetFeatureStore`、feature Parquet 读写和 manifest 校验。
- `configs/paper/*_parquet.toml`、`scripts/v1-smoke.ps1`、Binance/IBKR paper 脚本已经使用 Parquet 路径。

准确说法是：当前 Parquet 已覆盖本地 bars 和 feature store 边界，但还不是旧版 `docs/archive/legacy/database.md` 规划的完整分区化历史数据湖，例如 tick、order book、funding rate、open interest、fundamentals、corporate actions 等全量 schema。

### Workspace 当前实现

Cargo workspace 当前包含 2 个 app 和 24 个 crate。旧分析漏掉了 `algorithm` crate。

关键实现方向是成立的：

- `algorithm` 统一执行 Universe、Alpha、Portfolio、MarketRules、Risk、Execution、OMS。
- `storage` 负责 SQLite repository 和 event persistence。
- `data` 负责 CSV / Parquet bars。
- `feature_store` 负责 feature records、Parquet、manifest。
- `paper` 已有 simulated、Binance、IBKR paper 执行路径。
- `runtime` 管理 active run、cancel 和 live surface。

### 金融精度与账本

当前实现在核心路径中使用 `rust_decimal::Decimal` 表达价格、数量、金额、费用、保证金等字段，这是正确方向。

仍需注意：

- SQLite 中金额和数量以字符串保存是合理的，但所有边界转换必须集中在 storage/repository 层。
- IBKR adapter 和外部 SDK 边界可能需要 f64 字段转换，必须只在 adapter 边界发生，并保留 Decimal 领域模型。
- 合约 funding、margin mode、position side、leverage 已有 storage boundary；simulated paper 已形成 fill -> funding settlement -> `crypto_positions` 写入闭环。

## 关键风险

### 1. 合约持仓模型仍未达到生产完整度

当前代码已经识别 `CRYPTO_PERP` / `CRYPTO_FUTURE`，并有 short 权限、gross exposure、leverage、margin used、contract-specific leverage/margin/funding bounds 等基础风控逻辑。`crypto_positions` 和 `funding_rates` 表已作为 storage boundary 存在，并有 API/CLI 只读查询面。simulated paper runtime 已在合约成交和 funding settlement 后写入 `crypto_positions`，Binance position reconciliation 也已有 drift 检测测试。

- IBKR 合约 account/position snapshot adapter 已接入，live API 可按配置注入 IBKR paper Gateway；paper-account production reconciliation 已通过 6 轮 read-only、3 轮 paper order-submit、30 轮 extended read-only 验收，fresh multi-broker read-only reconciliation gate evidence 已闭环；完整 live production reconciliation 仍需扩展到真实资金账户、更多资产、更多 broker、filled orders 和更长运行窗口
- liquidation price / buffer 的真实交易所数据接入
- open interest / contract metadata 自动更新
- 跨交易所 portfolio margin

因此当前可以做 simulated contract accounting、funding settlement、storage round-trip、API/CLI readback、Binance reconciliation drift 检测、IBKR account/position snapshot broker-boundary readback、IBKR paper-account production reconciliation 验收，以及 paper/Testnet multi-broker read-only reconciliation gate 验收；但不能宣称具备完整合约实盘交易台账。完整支持还需要真实资金 live-account reconciliation 长跑、filled-order reconciliation、交易所 liquidation/metadata 数据和生产告警。

### 2. 市场规则已开始 runtime 装配，但仍不完整

`market_rules` crate 已经承担 lot size、tick size、min notional、初始保证金率等校验。`paper` runtime 现在会从 storage 读取 `lot_size_rules` / `price_limit_rules`，按 symbol 装配运行时规则并注入 algorithm；没有数据库规则时仍回退代码默认规则。simulated paper fills 也会读取 `fee_rules` 的 maker/taker bps、可选税费、可选交易所附加费和可选最低费用，并按订单类型推断 maker/taker 角色来计算模拟成交费用，真实 Binance/IBKR paper executor 仍保留 broker 回报原始费用。batch paper run 还会从 storage 预加载 `market_calendars` / `trading_sessions`，并通过 algorithm 的同步 trading schedule provider 在订单生成前拒绝休市日和非配置交易时段；stream paper run 也会在每个 slice 进入 algorithm 前按 symbol/trading day 动态刷新 storage-backed calendar/session。`lot_size_rules` / `price_limit_rules` / `fee_rules` 的 insert 和 effective-to 版本收口现在也会写入 `event_store`，形成 storage 层变更审计链路。`crypto_market_meta` 和 `corporate_actions_meta` 也已有只读 API 查询面，并且 reference-data ingestion 可以从 Binance/Yahoo 填充部分真实参考数据。这个选择适合 MVP，但它仍不是完整配置化市场规则系统。

当需要频繁调整市场规则、覆盖多个交易所、按时间版本化规则时，需要把这些表接入规则装配和变更流程：

- `fee_rules` 的阶梯费率模型
- `market_calendars` / `trading_sessions` 的更完整市场日边界处理
- 市场规则变更审批、发布和 runtime 动态刷新
- `crypto_market_meta`

### 3. 审计投影已存在但字段覆盖仍有限

`event_store` 是合理的最小审计真源，尤其适合保存 typed event payload 序列化后的完整事件。

当前已经新增 `order_events`、`risk_events`、`insights`、`portfolio_targets` 作为结构化查询投影，不再需要靠手工解析所有 JSON 才能按 run 查询核心审计记录。live 启动恢复现在会在 broker open orders/executions 与本地 recoverable orders 匹配成功后写入 `broker.order.recovered`，从而让订单恢复也进入 `order_events` 查询面。恢复会把已有本地 fills、新 broker executions 和本地订单已记录的 filled qty 作为累计下限，避免重启后丢失或回退半成交进度。未知远端 open order 默认会在写入启动恢复 WARN 诊断后升级为恢复失败，并通过现有边界把 run 标记为 failed；只有显式配置 `[live.startup_recovery] unmatched_open_orders = "warn_only"` 时才降级为告警继续运行。未知 execution 仍先保留 WARN，因为可能来自历史窗口或其它系统。Binance 恢复边界已处理 `CRYPTO:BINANCE:...` canonical symbol 到 REST native symbol 的转换，并保留 `isBuyer` 推导出的成交方向。

仍需注意：

- 投影表必须从 `event_store` 派生，不能成为第二套事实来源。
- runtime payload 里仍有字段缺省，例如部分策略名、账户 ID 需要后续从上游事件补齐。
- 运营查询已有本地端到端 smoke 覆盖 live snapshots、reconciliation、logs、config-version，以及 staging config 的 approval queue 和 release/audit readback；runtime 单测覆盖了启动恢复、未知远端 open order 默认阻断启动、显式 warn-only、未知 execution 告警和恢复失败停机；IBKR paper-account production reconciliation 已完成 39 次 audit 验收，fresh multi-broker read-only reconciliation gate evidence 也已闭环，但仍未覆盖真实资金 live account、更多 broker / 资产和 filled-order 生产场景。

建议把 `event_store` 保持为不可变审计真源；运营查询只读取 `order_events`、`risk_events` 等投影表，不要把它们设计成第二套真源。

### 4. 配置快照已有 API 运行快照，本地生命周期已接通

`strategy_runs.config_json` 能保存运行配置快照，API 启动的 Backtest、Paper 和 Replay 当前会把本次 TOML 配置解析为结构化 JSON 写入运行记录；API 启动 Backtest/Paper/Replay/Live 和 CLI 启动 Backtest/Paper/Replay 都会把原始 TOML 以 `RUN` 配置快照写入 `configs` 表，并绑定到 `run_config_versions`，足够支撑本地运行的可复现查询。

独立 `configs` 表已经落库，并承载两类语义：API 运行配置快照，以及管理型配置版本。管理型配置现在支持 `draft -> pending_review -> approved -> published -> archived` 状态机、版本列表、latest/published/specific 查询、target_env/rollout 元数据、production 发布独立审批约束、轻量 production role policy、pending approval queue、JSON diff、rollback 创建新 draft、状态变更写 `config_audits`/`config_releases`/`event_store`，并有 API/CLI 测试覆盖。它的完整生产价值还在于：

- 把配置版本和审批状态接入真实身份认证和 RBAC 权限系统。
- 绑定策略发布。
- 支持多用户、多环境权限矩阵、多人审批队列和生产变更审计报表。

如果项目仍处于单机本地验证阶段，暂时不需要急着把运行时配置加载全部迁到 `configs` 表。当前已经有配置版本、状态机、production 独立审批发布约束、轻量 role policy、pending approval queue、API/CLI run config snapshot、发布绑定和回滚审计的本地查询语义；真正需要补的是带身份认证的 RBAC、多环境权限矩阵和多人审批队列。

## 对旧分析的修正

| 旧说法 | 修正后说法 |

| 旧报告关于 Parquet 缺失的说法 | Parquet 已实现 bars 和 feature store；完整历史数据湖未实现 |
| 固定测试数量和通过比例 | 不应写死；以当前 CI / `scripts/verify` / `cargo test` 实际结果为准 |
| event_store 替代多个事件表，所以 order_events 不需要 | event_store 可作为审计真源；order_events 可作为查询投影，是否需要取决于生产查询要求 |
| positions 通过 asset_class 统一 crypto_positions | 当前通用 positions 表没有 asset_class；合约持仓使用独立 `crypto_positions` storage boundary，paper runtime 已接线 |
| 小规模回测不需要 Parquet | 当前已经使用 Parquet；问题是是否需要分区化历史数据湖 |
| 数字货币合约 PnL 已能靠现有表处理 | simulated paper 已有 realized/unrealized/funding PnL 写入；真实 broker 对账和生产级合约元数据仍不完整 |

## 建议路线

### 短期：保持文档和实现对齐

- 明确旧版 `docs/archive/legacy/database.md` 的 24 张目标表已经有 migration/storage boundary，但不等于 runtime 全部自动写入；当前实现说明以 `docs/tech/storage.md` 为准。
- 持续用 `scripts/schema-gap-check.ps1` 防止目标表和 migration 再次漂移。
- 删除或改写所有把 Parquet 当作完全缺失的陈旧描述。
- 不再写死测试数量，改为引用验证脚本和 CI 结果。

### 中期：补审计投影

优先补和交易排障直接相关的结构化投影：

- `order_events`：订单状态变更、broker order id、外部回报、错误。
- `risk_events`：风控输入、阈值、拒绝原因、目标仓位。
- `insights` / `portfolio_targets`：策略信号和目标仓位的事后分析。

这些表应由事件或运行结果写入，不能绕过 `event_store` 成为另一套事实来源。

### 中期：补市场规则配置

如果要支持多个股票市场或交易所，需要逐步把硬编码规则变成可版本化配置：

- 交易日历和交易时段。
- 手续费与滑点模型。
- lot size、tick size、min notional。
- 涨跌停和特殊交易限制。

### 长期：补合约与生产账本

如果要做生产级 crypto perp / future，下一步必须补齐真实 broker 和 reference-data 生产运维链路：

- 更多 broker / 多交易所完整 position reconciliation 长跑验证
- 非 IBKR broker reported cash/account snapshot 调度
- 更多真实 broker 的 open orders/executions 启动恢复联调和长跑验证
- open interest / liquidation buffer 等尚未覆盖的交易所参考数据采集
- reference-data 限流退避、陈旧数据告警和生产重试策略
- liquidation price / buffer 数据接入
- production alerting

没有这些能力前，不应宣称支持完整数字货币合约实盘。

## 最终判断

当前系统是正常的渐进式 MVP 架构：核心链路清晰，模块拆分合理，SQLite 已覆盖 24 张目标表和 `event_store`，足以支撑本地回测、Replay、paper run、报告、审计事件和 storage-boundary 验证；Parquet 与 feature store 也已经进入实现。

但它不是完整生产交易系统。真正的差距不在于“表数量少”本身，而在于几个金融系统必须严肃处理的能力尚未完整落地：

| 差距项 | 当前状态 | 已完成证据 | 剩余工作 | 下一步 |
| --- | --- | --- | --- | --- |
| 结构化订单与风控审计 | partial | `event_store`、`order_events`、`risk_events`、启动恢复 `broker.order.recovered` 投影、reconciliation drift 风险事件、run-scoped audit/API/CLI 查询面；IBKR paper production reconciliation 39 次 audit evidence 和 fresh multi-broker read-only reconciliation gate evidence 已通过 | 继续补齐 runtime payload 字段覆盖；扩展到真实资金 live account、更多 broker、更多资产、filled orders 和更长窗口的 reconciliation 审计验证 | 扩展 filled-order / live-money reconciliation evidence |

## fee_rules 阶梯费率小切片设计

本次只落 fee tier 数据模型、storage round-trip 和 simulated paper 的本 run 成交额聚合，不引入完整账户级成交量状态。

- 阶梯维度：tier 归属到一条 `fee_rules`，继承 `market/exchange/asset_class/effective_from_ms/effective_to_ms`。当前不按账户区分，不表达滚动 30d 或自然月窗口；这些属于后续账户级 volume state。
- volume 来源：simulated paper 用本 run 已成交 notional 累计量选择下一笔 fill 的 tier。真实 broker paper、broker/account snapshot 和跨 run 聚合暂不参与。
- fee rule 匹配顺序：runtime 按 symbol 解析出的 `market/exchange/asset_class` 找当前有效 `fee_rules`，顺序是 `symbol` 精确匹配、`asset_class` 通用规则、`asset_class='*'` 的 exchange default；tier 只在命中父 rule 后生效。
- 费用组成：tier 只覆盖 maker/taker bps；`tax_bps`、`exchange_fee_bps`、`minimum_fee` 仍来自父 `fee_rules`，并按“maker/taker + tax + exchange 后再应用 minimum floor”计算。
- 边界：回测和 simulated paper 可共享这个纯计算模型；真实 IBKR/Binance paper executor 继续使用 broker 回报 fee，不用本地 tier 覆盖 broker 成交回报费用。
